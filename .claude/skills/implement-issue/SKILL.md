# Issue実装ワークフロー（TDD）

指定されたGitHub Issueの修正方針に従い、TDD（テスト駆動開発）でコードを実装する。

## 前提

- 引数 `$ARGUMENTS` にIssue番号またはIssue URLが渡される
- Issue URLの場合は末尾の番号を抽出してIssue番号として使用する
- 引数がない場合はユーザーにIssue番号を確認する

## 手順

### 1. Issue情報の取得

```bash
gh issue view <Issue番号> --json title,body,number,labels,comments
```

Issueの **本文（body）とコメント（comments）の両方** から以下を把握する:
- **概要**: 何を修正するか
- **修正方針**: 具体的な修正手順
- **対象ファイル**: 変更が必要なファイル一覧
- **テスト観点**: テストで確認すべきポイント

**重要**: コメントには質問事項への回答や追加の要件・方針変更が含まれていることがある。コメントの内容がIssue本文と矛盾する場合は、より新しいコメントの内容を優先する。

### 2. 対象コードの確認

Issueの「対象ファイル」に記載されたファイルを全て Read ツールで読み込み、現状のコードを理解する。

- 修正対象のメソッドや処理の前後のコンテキストを把握する
- 関連するファイル（テストファイル等）も確認する
- Issueの「現状」セクションのコードスニペットと実際のコードに差異がないか確認する

差異がある場合（Issueが古い、他のPRで変更されている等）はユーザーに報告し、方針を確認する。

### 3. RED: 失敗するテストを先に書く

#### 3a. テストの作成

Issueの修正方針・テスト観点に基づいてテストを作成する。

- 既存のテストファイルがあればそれに追加する
- 新規作成の場合は、既存テストの命名規則・構造に倣う
- CLAUDE.md にテスト作成の規約やパターンが記載されていれば必ず従う

#### 3b. RED確認

テストを実行して **失敗する** ことを確認する。

- テストが失敗（RED）すれば次のステップへ進む
- テストが成功してしまう場合は、テストがバグ/要望を正しく再現できていないので見直す
- テスト自体にエラー（シンタックスエラー等）がある場合は修正してから再実行する

### 4. GREEN: テストが通るようにコードを修正する

Issueの修正方針に従い、対象ファイルを Edit ツールで修正する:

- テストが通る **最小限** の修正を施す
- 過度なリファクタリングは行わない

修正後、テストを再実行して **成功する** ことを確認する。

- 全テストがGREEN（成功）になれば次のステップへ
- 失敗するテストがあれば修正を続ける

### 5. REFACTOR（必要な場合のみ）

テストがGREENの状態を維持しながら、明らかに改善が必要なコードがあれば軽微なリファクタリングを行う。

### 6. フォーマッター実行

CLAUDE.md にフォーマッター/リンターの実行コマンドが記載されていれば、修正したファイルに対して実行する。

### 7. 最終確認

関連するテスト全体を実行して、既存テストが壊れていないことを確認する。

### 8. 完了報告

ユーザーに以下を報告する:

```
## 実装完了サマリー

**Issue:** #<番号> <タイトル>

### 作成/修正したファイル
- `ファイルパス`: 変更内容の説明

### テスト結果
- テストクラス: XxxTest
- テスト数: N件
- 結果: 全件GREEN

### 次のステップ
`/create-pr` でPRを作成してください。
```
